# Signing Key Rotation Runbook

**Scope:** the PAdES signing key used by the calibration signing service.
Production keys live in cloud KMS/HSM (AWS KMS or Azure Key Vault); they are
never exportable and never exist on a device or in the repo.

## Key inventory

| Key | Where | Purpose | Rotation |
|---|---|---|---|
| Certificate signing key | KMS/HSM, non-exportable | PAdES signatures on issued certificates | 12 months, or immediately on suspicion of compromise |
| Internal CA key (v1) | Offline / KMS | Issues the signing certificate | 3–5 years |
| TSA | External trusted authority | RFC 3161 timestamps | n/a (external) |

## Provisioning (AWS KMS — implemented, issue #24)

One-time setup for the production signing path
(`backend/app/signing/keys.py`, `AwsKmsKeyProvider`):

1. **Create the key** (non-exportable, sign/verify usage):
   ```
   aws kms create-key --key-spec RSA_3072 --key-usage SIGN_VERIFY \
     --description "Prowalco certificate signing"
   aws kms create-alias --alias-name alias/prowalco-signing \
     --target-key-id <KeyId from above>
   ```
2. **Create an IAM user/role for the signing service** with a least-privilege
   policy — only these two actions, only on this key:
   ```json
   { "Version": "2012-10-17", "Statement": [{
       "Effect": "Allow",
       "Action": ["kms:Sign", "kms:GetPublicKey"],
       "Resource": "<key ARN>" }] }
   ```
3. **Issue the signing certificate** for the KMS public key from the internal
   CA (created on first run — keep `ca-key.pem` offline):
   ```
   cd backend && python scripts/issue_cert_for_kms_key.py \
     --key-id alias/prowalco-signing --region af-south-1
   ```
4. **Configure the signing service** (e.g. Render env) with the values the
   script prints:
   `SIGNING_KEY_PROVIDER=aws_kms`, `AWS_KMS_KEY_ID`, `AWS_REGION`,
   `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` (the step-2 principal),
   `SIGNING_CERT_PEM_B64`, `SIGNING_CERT_CHAIN_PEM_B64`.
   Remove `SIGNING_KEY_PEM_B64` — with aws_kms the private key never exists
   as a file. Startup fails fast if the key ID or certificate is missing.
5. **Verify**: sign a staging certificate; the signature must validate against
   `ca-cert.pem` (Adobe shows trusted once the CA is added to trust — the
   out-of-the-box green tick requires an AATL CA, tracked separately).
6. Every `kms:Sign` call is logged in CloudTrail — this is the independent
   signing log referenced in the e-signature procedure.

## Scheduled rotation (no compromise)

1. Create the new asymmetric key in KMS (`kms:CreateKey`, signing usage).
2. Issue a new signing certificate for it from the internal CA (v1) or the
   public CA (later), with `digitalSignature` + `contentCommitment` usage.
3. Deploy the new key ID / certificate chain to the signing service
   configuration (`SIGNING_KEY_PROVIDER` config; see
   `backend/app/signing/keys.py`).
4. Verify: sign a test certificate in staging; validate the signature chain
   (green tick in Adobe Reader against the CA).
5. Cut over. Do **not** delete or disable the old key.
6. Mark the old key "verify-only": previously issued certificates must remain
   verifiable for the full retention period (≥ 5 years). Keep the old
   certificate published.
7. Record the rotation (date, key IDs, operator) in the quality system.

## Emergency rotation (suspected compromise)

1. Immediately disable signing with the suspect key (KMS key policy — deny
   `kms:Sign`); the signing service fails closed and queued packages remain
   safely queued on devices.
2. Follow scheduled-rotation steps 1–5 with a freshly issued certificate.
3. Determine the window of possible misuse from KMS CloudTrail / Key Vault
   logs, and reconcile every signature in that window against the append-only
   audit trail (each issue event records the signed PDF SHA-256).
4. Certificates confirmed legitimate stand; any signature not matching an
   audit record is treated as forged — notify the QM and affected customers,
   and revoke the signing certificate via the CA (CRL/OCSP).
5. Write an incident report; review whether TSA timestamps bound the window.

## Notes for the PoC

- Dev/test uses a local self-signed key generated by
  `backend/scripts/generate_dev_signing_cert.py` into the gitignored
  `backend/dev-keys/` — clearly marked NOT FOR PRODUCTION.
- The KMS provider is a stub (`AwsKmsKeyProvider`) documenting the intended
  integration (pyHanko `ExternalSigner` + `kms:Sign` on message digests).
